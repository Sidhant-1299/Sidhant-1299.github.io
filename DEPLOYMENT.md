# Deployment

## Current Status
- This portfolio is prepared for deployment, but the repository is not actively deployed yet.
- GitHub Pages is the likely first target because this repo is named `Sidhant-1299.github.io`.
- Keep deployment-specific decisions in this file so future hosting changes can be made quickly without searching through app code.

## GitHub Pages
- For the user site `Sidhant-1299.github.io`, keep Vite's base path as `/`.
- The GitHub Actions workflow in `.github/workflows/deploy.yml` builds the app, uploads `dist`, and deploys with `actions/deploy-pages`.
- The workflow copies `dist/index.html` to `dist/404.html` so direct route visits can fall back to the React app.
- Use `npm ci` in CI because `package-lock.json` exists and should be treated as the dependency source of truth.

## Project Pages
- Only use this mode if the site is deployed under a GitHub Pages subpath such as `https://user.github.io/repo-name/`.
- In that case, update `vite.config.js` with `base: '/repo-name/'`.
- Keep the same SPA fallback step by copying `dist/index.html` to `dist/404.html`.

## Static Hosts
- Netlify, Vercel, Cloudflare Pages, and similar hosts should run `npm run build` and publish `dist`.
- Keep Vite's base path as `/` for root deployments and custom domains.
- Configure the host to rewrite all non-file requests to `/index.html`.
- If the host supports a redirects file instead of dashboard rewrites, use its standard SPA fallback rule.

## VPS Or Static Server
- Build with `npm run build` and serve the generated `dist` directory.
- Keep Vite's base path as `/` when serving from the domain root.
- Configure the server to return `/index.html` for routes that do not match a real static file.
- For Nginx, the equivalent behavior is `try_files $uri $uri/ /index.html` inside the site location block.

## Vite Base Rules
- Use `/` for `Sidhant-1299.github.io`, custom domains, Netlify, Vercel, Cloudflare Pages, and root VPS deployments.
- Use `/repo-name/` only for GitHub Pages project-subpath deployments.
- Do not change `base` during routine portfolio work. Change it only when the hosting target changes.

## SPA Routing
- The app uses History API routes in `src/App.jsx`.
- These routes must work after direct browser refreshes: `/work`, `/academic`, `/about`, and `/projects/[slug]`.
- Hosts that do not support rewrites need a `404.html` fallback that contains the built app.
- GitHub Pages uses the `dist/404.html` fallback generated by the deploy workflow.

## Pre-Deploy Checklist
- Run `npm ci` from a clean checkout or CI environment.
- Run `npm run lint` and resolve deployment-blocking issues.
- Run `npm run build` and confirm Vite completes successfully.
- Preview the built app when possible with `npm run preview`.
- Check desktop and mobile layouts for `/`, `/work`, `/academic`, `/about`, and at least one `/projects/[slug]` page.
- Refresh direct routes such as `/work` and `/projects/semantic-book-recommender` on the target host to confirm fallback behavior.

## Rapid Change Notes
- Keep hosting changes isolated to `.github/workflows/deploy.yml`, `vite.config.js`, host rewrite files, and this document when possible.
- Prefer changing deployment configuration over changing route code for host-specific behavior.
- Preserve the route contract from `AGENTS.md`: do not add `/archive`, and keep the primary nav as Home, Work, Academic, About.
- If a future host requires a different fallback mechanism, document it here before or alongside the config change.
